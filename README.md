# 改修プロジェクトのための 12 の質問

---

## このドキュメントの目的

改修対象である既存システムについて、調査担当者が**「答えられるべき 12 の質問」**とその回答テンプレートをまとめたものです。

これらの質問に答えられる状態になることで、

- 改修プロジェクトの SS(システム要件定義)の「現行システム」章のインプットが揃う
- チーム全員が同じ現状理解を持った状態で PG 工程に入れる
- 改修時の地雷(触ると壊れる箇所、属人化した運用)を事前に発見できる

ことを狙います。

---

## このドキュメントの使い方

1. 12 の質問を **この順番** で進めてください(後ろの質問は前の質問の理解を前提にしています)
2. **広く浅く一周してから深める**: Q1 から完璧にしようとせず、まず全質問にざっと回答してから、重要な箇所を深堀りしてください
3. 各質問には **回答欄** と **「分からなかったこと」欄** があります。両方埋めてください
4. **分からなかったことを書くのは失敗ではなく価値です**。空白で誤魔化さず、「ここまで分かった / これは分からなかった / 次に聞く相手はこの人」と正直に記録してください

---

## 大原則

### 完璧を目指さない

各質問は「分かったこと」「分からなかったこと」「次の調査の手がかり」の 3 つが書けていれば合格です。全ての細部を埋める必要はありません。

### 2 人でまず相談する

詰まったら、まずペアで 5〜10 分相談してから、それでも分からなければ責任者(SS 担当)に質問してください。質問は即時ではなく Teams 等に溜めて、定期的にまとめて回答する運用にします。

### コードに書いていないことを意識する

業務ルール、手動運用、属人化したノウハウ、過去のトラブル履歴などは、**コードを読んでも出てきません**。必ず人にヒアリングする時間を取ってください。早めにヒアリング相手を特定して、アポを取りに行くのがコツです。

### 情報源を必ず記録する

「誰から聞いたか」「どのドキュメントを見たか」「どのコードを読んだか」を残してください。後で「これって本当?」と確認するときの根拠になりますし、SS のレビュアーから出典を聞かれたときに即答できます。

---

## 進め方の目安(2 週間)

| 期間 | フォーカス | 補足 |
|---|---|---|
| 1 週目 前半 | Q1〜3(目的・境界)、Q4〜6 に着手 | 全体像をざっとつかむ |
| 1 週目 後半 | Q4〜6(構造)に集中 | 図を描く時間を取る |
| 2 週目 前半 | Q7〜10(ドメイン・運用・検証) | 必要なヒアリングを実施 |
| 2 週目 後半 | Q11〜12(リスク)、全体整合性チェック | 「分からなかったこと」を整理 |

# 付録: ヒアリング時のコツ

- **「分からない」を素直に言う**: 知ったかぶりは情報の質を下げます
- **メモは録音より手書き / タイピング**: 聞きながら整理することで理解が深まる
- **「なぜそうなっているか」を必ず聞く**: 表面の事実だけでなく背景の理由を聞くと、改修時の判断材料になります
- **「過去に問題が起きたことはありますか」を必ず聞く**: Q11・Q12 の最大の情報源
- **「他に誰に聞くといいですか」を最後に必ず聞く**: 次のヒアリング相手が芋づる式に見つかります
